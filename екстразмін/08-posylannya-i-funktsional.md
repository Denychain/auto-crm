# 1. ПОРОЖНІ ПОСИЛАННЯ (`href="#"`)

## L-01. Логотип на landing → `<a href="#">`
**Файл:** `app/(site)/page.tsx:78`
```tsx

```
**Слідство:** клік перезавантажує сторінку через якорну прокрутку до верху,
без оновлення URL.
**Виправлення:** замінити на `<a href="/">` (як на інших сторінках).

## L-02. Політика приватності і Умови сервісу
**Файл:** `app/(site)/page.tsx:1321-1322`
```tsx
<a href="#">Політика приватності</a>
<a href="#">Умови сервісу</a>
```
**Слідство:** клік нічого не робить.
**Виправлення (на вибір):**

  з реальними текстами (потрібна юридична консультація).
- Або тимчасово прибрати посилання, бо порожні юридичні лінки гірші за відсутні.

## L-03. Картки галереї `href="#"` (sb на /gallery)
**Файл:** `app/(site)/gallery/page.tsx:397, 424`
```tsx

```
**Слідство:** кожна картка галереї — це посилання, яке не веде нікуди.
Користувач очікує модалку з повним розміром фото, або сторінку з деталями
проєкту.
**Виправлення:**
- Створити lightbox (модалку) з повним зображенням і описом проєкту.
- АБО створити окрему сторінку `/gallery/[id]/page.tsx` з детальним кейсом.
- АБО, якщо рішення про детальний перегляд не прийняте — змінити елемент

# 2. ПОСИЛАННЯ З НЕТИПОВИМИ ПРОБЛЕМАМИ

## L-04. Анкорна навігація на landing замість окремих сторінок

Поточно на landing нав-бар:
```tsx
<a href="#hero">Головна</a>
<a href="#us-cars">Авто з США</a>   {/* секція на landing */}
<a href="/gallery">Галерея</a>      {/* окрема сторінка */}
<a href="#about-master">Про майстра</a>  {/* anchor в пустоту */}
<a href="#contacts">Контакти</a>    {/* footer */}
```

**Інконсистентність:** деякі ведуть на anchors, деякі — на сторінки. Окрім того,
є окремі сторінки `/us-cars`, `/contacts`, `/master`, але landing на них **НЕ
лінкує** з основного нав-бару. Користувач не дізнається про існування цих
сторінок з landing.

**Виправлення (стратегічне):**
Стандартизувати — нав-пункти ведуть на сторінки:
```tsx
<a href="/">Головна</a>
<a href="/us-cars">Авто з США</a>
<a href="/gallery">Галерея</a>
<a href="/master">Про майстра</a>
<a href="/contacts">Контакти</a>
```
**На landing** секція `#us-cars` лишається як тизер — з кнопкою «Дивитись
детальніше → /us-cars». Те саме з контактами і майстром.

## L-05. Анкор `#about-master` веде в пустоту
**Файл:** `app/(site)/page.tsx:213-214` (порожній span перед `philosophy`).
Це симптом проблеми L-04. Рішення там само — лінкувати на `/master`.

## L-06. Viber-посилання `viber://chat?number=...` працює лише на мобільних
**Файл:** `lib/constants.ts:6`
```ts
viber: "viber://chat?number=%2B380992334420"
```
**Поведінка:** на десктопі без встановленого Viber-клієнта браузер
показує помилку «протокол не знайдений». На Android/iOS — відкриває Viber
з номером.
**Виправлення:** додати fallback. Один з варіантів:
```ts
viber: "https://msng.link/o?380992334420=vi"
```
Цей сервіс редиректить на нативний Viber на мобільних, на десктопі показує
«встановіть Viber». Або власне обгортка-сторінка `/open/viber`.

## L-07. Telegram-посилання вимагає публічний username
**Файл:** `lib/constants.ts:5`
```ts
telegram: "https://t.me/+380992334420"
```
**Поведінка:** Telegram **не відкриває чат з номером** через web-URL, якщо
у користувача немає публічного username. `https://t.me/+380...` працює
лише в певних випадках (для contacts уже доданих в адресну книгу).
**Виправлення:**
- Якщо у Дмитра є публічний username (напр. `@nicecarif`) — використати
  `https://t.me/nicecarif`.
- Інакше — `tg://resolve?phone=380992334420` (deep-link, працює на пристроях
  з Telegram).
- Найкраще — обидва варіанти у різних місцях: head лінкує на публічний
  username (видимий), а форма заявки — на deep-link.

## L-08. Waze/Apple Maps на /contacts захардкоджені координати
**Файл:** `app/(site)/contacts/page.tsx:373, 382`
```tsx
href="https://waze.com/ul?ll=48.9226,24.7111&navigate=yes"
href="https://maps.apple.com/?q=48.9226,24.7111"
```
**Слідство:** координати `48.9226, 24.7111` — задаються тут жорстко. Якщо
адреса в `CONTACTS` зміниться, ці посилання застаряють.
**Виправлення:** додати координати в `CONTACTS`:
```ts
coordinates: { lat: 48.9226, lng: 24.7111 }
```
і використати у URL.

**Окремо:** ці координати ПОТРІБНО перевірити з власником. Зараз вони
співпадають з map placeholder у footer landing (рядок 1292-1293), але я
не перевіряв, чи це справді координати цеху на вул. Максимовича, 15.

# 7. Що увійде в пояснювальну записку

Цей файл — основа підрозділу **«Інтегральність користувацького інтерфейсу
і функціонального покриття»**:

- **Аудит UX-цілісності** (розділ 1-2) — методологія перевірки кожного
  інтерактивного елементу. Демонструє підхід до якості як до контрактного
  зобов'язання.
- **«Технічний борг» vs «нереалізована функціональність»** (розділ 3) —
  розрізнення між «зламано» і «ще не написано». Сильний матеріал для
  розділу планування.
- **Стратегія deprecation для placeholder-функціоналу** (F-01) — як
  не обманювати користувача, поки backend ще в розробці. Тимчасові
  «правдиві» fallbacks замість фейкового success.
- **Single source of truth для навігації** (розділ 6, крок 2) — `SITE_NAV`
  як приклад data-driven UI. Один масив керує всіма меню.
- **Тестова матриця посилань** (розділ 5) — формат таблиці «рядок |
  href | стан | коментар» як шаблон для регресійного тестування
  навігації.
