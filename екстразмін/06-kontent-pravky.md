# 1. Фактичні розбіжності з реальними даними

## C-01. Графік роботи відрізняється від підтвердженого власником

**Файл:** `lib/constants.ts:13-17`
**Зараз у коді:**
```ts
hours: {
  weekdays: "09:00 — 18:00",
  saturday: "09:00 — 14:00",
  sunday: "Вихідний",
}
```
**Підтверджено власником:** Пн–Пт **10:00–18:00**, Сб **вихідний**, Нд вихідний.

**Що змінити:**
```ts
hours: {
  weekdays: "10:00 — 18:00",
  saturday: "Вихідний",      // не 09:00 — 14:00
  sunday: "Вихідний",
}
```

**Де ще «впишеться»:** автоматично у footer landing (рядок 1260-1263),
у /contacts (рядок 13-21, масив DAYS), у /gallery footer (рядки 553-555),
у /us-cars footer (715-717). Усі читають з `CONTACTS.hours`, отже одна
правка — і всюди узгоджено.

**ОДНАК** у `app/(site)/gallery/page.tsx:553-557` і `us-cars/page.tsx:714-717`
графік ХАРДКОДЖЕНИЙ:
```tsx
<li>Пн–Пт · 09–18</li>
<li>Сб · 09–14</li>
<li>Нд · вихідний</li>
```
Це треба замінити на `CONTACTS.hours.*` у всіх трьох місцях,
щоб був один джерело правди.

## C-02. На сторінці /contacts логіка «відкрито/зачинено» не врахує суботу

**Файл:** `app/(site)/contacts/page.tsx:72-81`
```ts
const open =
  (d >= 1 && d <= 5 && hh >= 9 && hh < 18) ||
  (d === 6 && hh >= 9 && hh < 14);
```

Якщо субота тепер вихідна, друга частина умови (`d === 6 && hh >= 9 && hh < 14`)
залишить субота показуваною як "відкрито" — баг.

**Виправлення:** прибрати другу частину, лишити лише будні. Або взагалі
винести логіку в `lib/hours.ts` і обчислювати з `CONTACTS.hours`.

## C-03. Конфлікт «років досвіду» між сторінками

| Сторінка   | Твердження                              |
|------------|------------------------------------------|
| Landing    | «25 років у ремеслі» (rep-stats 967)    |
| Master     | Journey від 2001 → 2026 = 25 років       |
| Gallery    | «12 років архіву фото» (counters 278)   |
| US-cars    | (не вказано)                             |

12 років архіву vs 25 років досвіду — формально не конфлікт (архів фото
почали вести пізніше), але виглядає двозначно. Або уточнити:
**«Архів фото · 2014 → 2026»** = 12 років з моменту заснування цеху (співпадає
з journey 2014).

**Що змінити:** в `app/(site)/gallery/page.tsx:278` — «12 років архіву фото»
→ **«12 років цеху · 25 років майстра»** або просто «з 2014 року».

## C-04. На landing блок «Про майстра» — це anchor у пустоту

**Файл:** `app/(site)/page.tsx:213-214`
```tsx
<span id="about-master" style={{ display: "block", visibility: "hidden", height: 0 }} aria-hidden={true} />
<section className="philosophy" id="philosophy">
```

Клік «Про майстра» у мобільному меню → потрапляє на anchor `#about-master` →
а це порожній span перед секцією «ФІЛОСОФІЯ». Тобто користувач очікує
розповідь про Дмитра, а бачить філософію цеху.

**Що змінити:** один з двох варіантів —
**(а)** прибрати anchor з landing, лінкувати «Про майстра» **на сторінку `/master`**
(вона вже існує і змістовна).
**(б)** додати на landing справжню секцію «Про майстра» з фото Дмитра і
коротким біо + кнопкою «Дізнатись більше» → `/master`.

Я рекомендую **(а)** — менше дублювання, /master сторінка вже багата.

## C-05. Інконсистентна навігація між сторінками

| Сторінка     | "Про майстра" веде на        |
|--------------|--------------------------------|
| `/` (landing)| `#about-master` (anchor)       |
| `/us-cars`   | `/#about-master`               |
| `/gallery`   | `/#about-master`               |
| `/master`    | `/master` ✓                    |
| `/contacts`  | `/#about-master`               |

Якщо обираємо варіант (а) з C-04 — у всіх місцях має бути `/master`. Зараз
лише /master сторінка сама себе так лінкує.

Так само мобільні меню. У всіх п'яти сторінках мобільне меню має бути
ідентичним — це означає винести його в окремий компонент `<MobileMenu />`
у `components/site/`. Зараз воно скопійоване 5 разів з різними хрефами.

## C-06. Інконсистентний хедер між сторінками

Те саме що C-05 — header копіпаст × 5. Винести в `<SiteHeader />`.
Особливо помітно:

- На landing логотип має `<a href="#">` (рядок 78) — клік перезавантажує
  сторінку. На інших сторінках `<a href="/">` — правильно.
- На landing нав-пункт «Контакти» веде на `#contacts` (anchor у футер).
  На інших — на `/contacts` (окрема сторінка). Якщо є /contacts — логічно
  скрізь вести на неї.
- На landing нав «Авто з США» — `#us-cars` (секція). На інших — `/us-cars`
  (сторінка). Та сама дилема: якщо є дві окремі сторінки — секція на landing
  має бути тизером з кнопкою «Дивитися детальніше → /us-cars», а нав-пункт
  лінкує на сторінку.

## C-07. «Адміністрація не несе відповідальності за цінні речі»

**Файл:** `app/(site)/page.tsx:1309`
> «Адміністрація не несе відповідальності за цінні речі, залишені в салоні.»

Юридично проблемне формулювання. По-перше, в Україні таке відмовлення
може не діяти за ЗУ «Про захист прав споживачів». По-друге, репутаційно
звучить як заяву великого СТО, а не майстерні «робимо як для себе».

**Що змінити:** Або прибрати, або переформулювати в стилі «Просимо
**забрати цінні речі з салону** перед передачею ключів» — це робить
відповідальність двосторонньою і звучить ввічливо.

## C-10. SEO keywords block — приховаваний keyword stuffing

**Файл:** `app/(site)/page.tsx:1328-1330`
```tsx
<p className="seo-keywords" aria-hidden={true}>
  Кузовний ремонт Івано-Франківськ · рихтування авто · фарбування в камері · …
</p>
```

Це класичний **SEO black-hat 2010-х** — прихований текст з ключовими словами.
Зараз Google за таке штрафує. Краще — органічно вписати ключові слова в
звичайні параграфи (вони вже там здебільшого є), а цей блок прибрати.

## C-12. «Перетягніть лінію» на /gallery — підпис лишається коли не на десктопі

## C-13. Слова «халтура» / «вуйко» / «хочу-лист» — діалектизми

Сайт у Івано-Франківську, локальна мова це плюс. Але «вуйко» (на /master:
«цех мого вуйка») для не-галичан незрозуміло. **Підказка:** додати поряд
у дужках «(дядька)» або переписати «у дядька».

«Хочу-лист» на /us-cars FAQ Q.06 — пересічно нормально, але «wishlist»

«Халтура» (landing 1206 у slogan-ах) — у переносному сенсі норм, але
для бізнес-сайту трохи різко. Можна замінити на «недбала робота».

# 2. Структурні правки (не контент, але близько)

## C-15. Дублювання header / mobile menu / footer × 5

Зараз кожна сторінка містить локальну копію header (~30 рядків), мобільного
меню (~12 рядків) і футера (~50 рядків). При зміні навігації треба правити
в 5 місцях.

**Що зробити:** винести в `components/site/SiteHeader.tsx`, `SiteMobileMenu.tsx`,
`SiteFooter.tsx`. Передавати `activePage` як prop, щоб підсвічувати поточний
пункт навігації.

Альтернатива — використати **`layout.tsx`** у `app/(site)/layout.tsx`:
header/footer там → і автоматично загорнуть усі дочірні сторінки. Це
ідіоматичний Next.js App Router patterns.

# 3. Що увійде в пояснювальну записку

Цей файл — матеріал для розділу **«Контент-стратегія та якість даних
публічного сайту»**:

- **Аудит фактичних даних** (розділ 1) — методологія перевірки тверджень
  на конкретність і верифікованість. Сильний приклад того, як технічний
  виконавець ОБОВ'ЯЗКОВО перевіряє контент від замовника, а не сліпо
  переносить.
- **Single source of truth** (C-15) — патерн, коли всі дані (контакти,
  графік, гарантія) централізовані в одному модулі. Аргументація через
  cost of change.
- **Анти-патерн SEO 2010-х** (C-10) — приклад того, чому старі техніки
  стають шкодою, а не користю. Кейс для розділу про SEO.
- **Юридичний ризик контенту** (C-07, C-14) — приклад того, як один
  невдало сформульований рядок може створити проблеми.
- **Інконсистентність копірайту** (C-08, C-19) — приклад чому потрібен
  центральний документ копірайту перед розробкою.
