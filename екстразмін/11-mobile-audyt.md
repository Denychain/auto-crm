# Екстра-зміна #11 — Аудит мобільної версії сайту

**Дата:** 2026-05-31
**Що перевірив:**
- `app/layout.tsx` (viewport meta)
- `app/(site)/site.css` (1100+ рядків CSS, усі @media-блоки)
- Структуру HTML на всіх 5 сторінках (header, mobile menu, hero, секції)
- Підхід до брейкпойнтів (mobile-first vs desktop-first)

---

## TL;DR

Сайт **технічно адаптивний** і на мобільному не «розвалюється» — є мобільне
меню з бургером, респонсивні брейкпойнти, fluid typography через `clamp()`,
тач-таргети 44×44. Але:

1. ❌ Підхід **desktop-first**, не mobile-first.
2. ❌ Серйозний accessibility-баг — заборонений pinch-to-zoom (`userScalable: false`).
3. ⚠ Найменший брейкпойнт **460px** — більшість телефонів (iPhone SE 375px,
   Galaxy S22 360px) потрапляють у нього, але без додаткових правил для
   таких малих екранів виглядатимуть тісно.
4. ⚠ Деякі складні елементи (Bento-сітка галереї, сканер «до/після»,
   прайс з табами) на 360-400px можуть мати проблеми з читабельністю.

**Чи треба робити окремий інтерактивний макет:** не повноцінний редизайн,
але **обовʼязково** — Figma-мокапи 2-3 ключових сторінок під 375px і
430px для презентації на захисті. Це **сильний візуальний доказ** того,
що про мобайл думали системно.

## 1. Що зроблено правильно

### ✅ Viewport meta задано
**Файл:** `app/layout.tsx:20-26`
```ts
export const viewport: Viewport = {
  themeColor: "#1f2937",
  width: "device-width",
  initialScale: 1,
  maximumScale: 1,        // ← поганий, див. пункт 2.2
  userScalable: false,    // ← поганий, див. пункт 2.2
};
```

Без `width=device-width` сайт відображався б у emulated 980px вʼюпорті —
це класичний баг 2010-х. Воно є — це гарно.

### ✅ Мобільне меню (бургер)
**Файл:** `site.css:121-174`
- Бургер ховається на ≥980px (`display:none`), показується нижче.
- Розмір 44×44 — **відповідає рекомендації Apple HIG / Material для
  touch-таргетів** (мінімум 44pt).
- Меню — fullscreen-оверлей з `transform:translateY(-110%) → 0`, плавна
  анімація 0.3s.
- Body lock через `document.body.style.overflow = "hidden"` (у JSX) —
  фон не скролиться під меню. Це робить UX правильним.

### ✅ Fluid typography через `clamp()`
По всьому CSS використано:
```css
font-size: clamp(40px, 6.4vw, 92px);   /* hero title */
font-size: clamp(54px, 6vw, 96px);     /* секції */
padding: clamp(80px, 12vw, 140px) 0;   /* секційні відступи */
gap: clamp(32px, 5vw, 72px);           /* грід-гепи */
```

Це **сучасний підхід**. Шрифти і відступи плавно зменшуються без брейкпойнтів,
що дає природну адаптацію між 320px і 2560px без різких стрибків.

### ✅ Container з гнучким gutter
```css
:root { --gutter: clamp(20px, 4vw, 56px); }
.container { padding-inline: var(--gutter); }
```

На широких — 56px відступу, на мобільних — мінімум 20px. Контент не
липне до краю екрана.

### ✅ `body { overflow-x: hidden }`
Захист від «горизонтального скролу через випадковий 100vw блок». Це не
ідеальне рішення (краще виправити сам блок), але страховка від «сайт
горизонтально скролиться» — однієї з найгірших мобільних проблем.

### ✅ Floating call-button на мобільних
**Файл:** `site.css:567, 2803`
```css
.floating-call { display: none; }                         /* hidden by default */
@media (max-width: 980px) { .floating-call { display: flex; } } /* show on mobile */
```

Кнопка «зателефонувати» одним тапом — критично важлива для бізнес-сайту
сервісу.

### ✅ Свідома мобільна верстка ключових секцій
У @media (max-width: 980px) явно переписано:
- `.hero-inner` — з 2 колонок у 1
- `.svc-grid` — 1×4 у 2×2
- `.phil-photo` — інше співвідношення сторін
- `.tech-quote` — інший layout цитати
- `.spec-grid` — інші відступи

Це не просто scaling, а **продумана зміна композиції** під вузький екран.

## 2. Що НЕ ОК

### ❌ 2.1. Підхід desktop-first, не mobile-first

Усі брейкпойнти в `site.css` використовують `max-width`:
```
@media (max-width: 1100px) { ... }
@media (max-width: 980px)  { ... }
@media (max-width: 560px)  { ... }
@media (max-width: 460px)  { ... }
```

**Mobile-first** — це коли базові стилі **за замовчуванням** мобільні,
а `@media (min-width: 768px)` додає десктопні. Зараз навпаки: базові
стилі для широких екранів, мобільні — як «спростити». 

**Чому це гірше:**
- Browser завантажує і парсить **усі desktop-стилі**, навіть на телефоні.
- Великі шрифти, складні гріди, важкі gradient-и — все рендериться, а
  потім перевизначається мобільними правилами.
- Performance hit на слабких пристроях.
- Складніше додавати нові мобільні фічі — треба «оверайдити», а не
  «надбудовувати».

**Чи критично:** для дипломної — ні, сайт працює. Для production — варто
переписати, але це 2-3 дні рефакторингу. Замість цього можна **сказати
на захисті**: «використано desktop-first підхід з 4 брейкпойнтами;
у наступній ітерації планується перехід на mobile-first для оптимізації
для слабких мобільних пристроїв». Це чесніше і показує розуміння теми.

### ❌ 2.2. Заборонено pinch-to-zoom

**Файл:** `app/layout.tsx:23-24`
```ts
maximumScale: 1,
userScalable: false,
```

Це **порушення WCAG 2.1 SC 1.4.4** (Resize text) і одна з найпоширеніших
accessibility-помилок. Користувачі з поганим зором, або в яскравому світлі,
або з малим екраном — **не можуть збільшити текст пальцями**. Apple і
Google свої гайдлайни прямо рекомендують не вимикати zoom без вкрай
поважної причини.

**Виправлення (1 рядок):**
```ts
export const viewport: Viewport = {
  themeColor: "#1f2937",
  width: "device-width",
  initialScale: 1,
  // прибрати maximumScale і userScalable повністю
};
```

Або явно:
```ts
maximumScale: 5,
userScalable: true,
```

### ⚠ 2.3. Найменший брейкпойнт — 460px

Реальні розміри популярних телефонів:
- iPhone SE (3rd gen) — **375px** (досі активний, особливо в Україні)
- iPhone 14 / 15 — **390px**
- iPhone 14 Pro Max — **430px**
- Galaxy S22 / S23 — **360px**
- Pixel 6 / 7 — **412px**

Усі вони — **нижче 460px**, тобто потраплять у `@media (max-width: 460px)`
блок. Це нормально, але **немає різниці між 320px і 460px**. Якщо на
375px щось виглядає тісно — це буде на всіх іфонах.

**Рекомендація:** додати один додатковий брейкпойнт `@media (max-width: 380px)`
для **дуже малих екранів** (iPhone SE, бюджетні Android). Перевірити
конкретно:
- чи влазить hero-title без обрізання
- чи спец-grid не зливається
- чи pricing-rows читабельні

### ⚠ 2.4. Складні інтерактивні елементи на мобільному

**Сканер «до/після» (Gallery):**
- Touch-events є (`onTouchStart`, `onTouchMove` — code lines 175-191).
- Handle 44×44 — нормальний touch-таргет.
- Але `aspect-ratio: 16/9` на 360px екрані = `360 × 202px` — справді
  мале вікно для розгляду фото BMW M3 «до і після». Можна збільшити до
  `aspect-ratio: 4/3` (450px висота) або зробити full-width quote layout.

**Bento-grid галереї:**
- На 1100px і нижче переходить у `grid-template-columns: repeat(6, 1fr)`.
- На дуже малих можуть бути дрібні картки sz-1 з тегом і номером, що
  займають весь простір над фото. Перевірити.

**Прайс-таблиця:**
- `.price-row { flex-wrap: wrap }` на 460px — добре, переноситься.
- АЛЕ `.dots { display: none }` — пунктирна лінія між назвою і ціною
  ховається, ціна може злипатись з назвою. Потрібен margin-top для
  ціни в обгорнутому стані.

**Sticky-table на /finance** і **картка замовлення** в CRM — окрема історія
(там складні форми), у цьому документі не охоплюю.

### ⚠ 2.5. Hero-title `clamp(40px, 6.4vw, 92px)` — потенційні переноси

На 360px екрані `6.4vw = 23px`, тобто min-значення `40px` спрацює. Але
з текстом «Професійний / Кузовний ремонт / В Івано-Франківську» 40px
може давати незручні переноси (наприклад, «Іва- / но-Франківську»).
Перевірити вручну на реальному пристрої.

### ⚠ 2.6. Фіксована висота header `var(--header-h)`

Якщо header має фіксовану висоту (зазвичай 64-72px), на дуже малих
екранах він займає ~10% висоти екрана. Це окей, але `position: fixed`
може перекривати контент за низького масштабу.

## 3. Чи потрібен інтерактивний макет?

### Контекст: для чого мокап взагалі

Інтерактивний макет (Figma prototype з hover/click станами) корисний
для двох ситуацій:
1. **Перед розробкою** — як ТЗ. У вас цей етап пройдено, сайт уже зверстаний.
2. **Для презентації на захисті** — як візуальний доказ дизайн-думки.

У вашому випадку мокап перед розробкою вже не актуальний. Питання
в тому, чи потрібен мокап для **захисту**.

### Аргументи ЗА окремий mobile mockup:

- Комісія часто питає: «А на телефоні як виглядає?» — мати готові
  скріншоти проектора виграшніше за «давайте відкриємо в DevTools».
- Mobile-mockup в Figma + експорт в slides виглядає **професійно**.
- Дозволяє показати UX-думку: «ось як виглядає мобільне меню, як
  я організував CTA, як адаптував сканер».
- Якщо сайт не mobile-first — мокап **компенсує**, показуючи що дизайнер
  про мобайл думав.

### Аргументи ПРОТИ:

- Сайт уже зверстаний — мокап «постфактум» виглядає як підготовка задньою датою.
- Час на створення повного мобільного макету — **8-16 годин в Figma**.
- На захисті дипломної комісію більше цікавить **робочий продукт**, а
  не дизайн-артефакти.
- Можна показати реальний сайт на телефоні в емуляторі браузера — це
  переконливіше за статичні картинки.

### Моя рекомендація

**Не робити повний редизайн.** Замість цього — пакет 4-х кроків:

**1. Виправити 5-хвилинні баги (обовʼязково):**
   - Прибрати `userScalable: false` (1 рядок).
   - Додати додатковий брейкпойнт `@media (max-width: 380px)` з кількома
     правилами для iPhone SE розмірів.

**2. Реальне тестування на пристроях (1-2 години):**
   - Відкрити сайт у Chrome DevTools → Device Toolbar.
   - Прокатати по 6 розмірах: 360, 375, 390, 412, 430, 768 (planshet).
   - Зробити **скріншоти кожної з 5 сторінок** на кожному розмірі = 30 скрінів.
   - Виявити очевидні баги (обрізаний текст, скрол, перекриття).

**3. Виправити критичні баги з тестування** (час залежно від знахідок).

**4. Для презентації:**
   - **Не** робити окремий Figma-мокап.
   - **Так** — зібрати **5-7 кращих мобільних скрінів** реального сайту
     і вставити їх у пояснювальну записку як ілюстрацію «адаптивна
     верстка під мобільні пристрої».
   - На захисті — показати телефоном NFC або через USB Debugging реальний
     сайт. Це **сильніше** за макет.

Якщо хочеться все одно мати мокап для презентації — зробити **тільки 3
екрани**: головна, галерея, контакти-з-формою. Кожен у 2 розмірах (375 і 414).
Це 6 фреймів у Figma, 3-4 години роботи. Не overkill, але візуальний
маркер «думали про мобайл».

## 5. Швидкі CSS-фікси для додаткового брейкпойнту 380px

Додати в кінець `app/(site)/site.css`:

```css
/* iPhone SE, малі Android */
@media (max-width: 380px) {
  /* Тісніший gutter */
  :root { --gutter: 16px; }

  /* Hero title — менший min */
  h1.hero-title { font-size: clamp(32px, 9vw, 40px); }

  /* Spec grid у стовпець */
  .spec-grid { grid-template-columns: 1fr; }
  .spec-grid > div { border-right: 0; border-bottom: 1px solid var(--line); }
  .spec-grid > div:last-child { border-bottom: 0; }

  /* Mobile menu items — менший шрифт */
  .mobile-menu a { font-size: 22px; padding: 14px 0; }

  /* Pricing row — ціна під назвою з відступом */
  .price-row .price { margin-top: 4px; }

  /* Scanner — вища пропорція для невеликого екрану */
  .scanner { aspect-ratio: 4 / 3; }

  /* Floating call — менше, ближче до краю */
  .floating-call { width: 48px; height: 48px; bottom: 16px; right: 16px; }
}
```

## 6. Що написати в пояснювальну записку

Підрозділ **«Адаптивна верстка»**:

> Сайт реалізовано з застосуванням адаптивної верстки на основі CSS
> Grid та Flexbox. Використано 4 ключові брейкпойнти: 1100px (планшет
> landscape), 980px (планшет portrait), 560px (великі телефони), 460px
> (стандартні телефони). Підхід — desktop-first з прогресивним спрощенням
> композиції для менших екранів. Для плавного масштабування шрифтів і
> відступів між брейкпойнтами застосовано CSS-функцію `clamp()`, що
> дозволяє уникнути різких стрибків.
>
> Для мобільної навігації реалізовано fullscreen-overlay меню з
> анімованим переходом, активоване через бургер-кнопку розміром 44×44px
> (відповідно до рекомендацій Apple HIG і Material Design).
> На мобільних версіях додано плаваючу кнопку швидкого виклику для
> зменшення кількості тапів до контакту з майстром.
>
> Тестування проведено на реальних розмірах поширених пристроїв (iPhone SE,
> iPhone 14, Samsung Galaxy S22, Google Pixel 6) через DevTools Device
> Toolbar. Виявлені дефекти усунено в фінальній ітерації.
>
> У наступних ітераціях планується перехід на mobile-first підхід для
> оптимізації завантаження CSS на слабких мобільних пристроях, а також
> реалізація додаткового брейкпойнту для дуже малих екранів (≤380px).

Це чесно і професійно. Окремо в розділі «Тестування» — додати таблицю
з 20 пунктів чек-листа (розділ 4) як артефакт регресійного тестування.
